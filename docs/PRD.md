## 2. 产品概述

Hetu 是一款面向个人的 AI 增强知识管理工具，以 **笔记** 与 **对话** 作为两大并列的核心载体：用户既可以在沉浸式编辑器中编写 Markdown 笔记，也可以与大模型进行多轮对话，并将对话中的高质量内容自动整理沉淀为结构化笔记。产品强调简洁、快速、本地化，帮助用户高效地记录、思考、整理和复用知识。

### 2.1 核心价值

- **笔记 + 对话双引擎**：同一产品内同时承载「写笔记」和「与 AI 对话」两种核心工作流，并支持两者之间的双向流转。
- **沉浸式写作体验**：支持 Markdown 快捷键、实时预览、自动保存。
- **结构化知识管理**：笔记本 + 标签双维度组织，支持全文检索与语义检索。
- **AI 智能增强**：接入大语言模型（LLM）与 Embedding 模型，支持对话、智能摘要、内容续写、语义搜索；支持 MCP 和 Skill 扩展 AI 能力。
- **对话沉淀为笔记**：对话中的有价值内容可一键或自动整理为结构化笔记，纳入知识体系。
- **本地优先**：默认使用 SQLite 本地存储，数据完全私有；可选配置 PostgreSQL 实现跨设备。
- **桌面应用就绪**：前期以 Web 启动，后续可套桌面壳（如 Electron / Tauri）。

### 2.2 目标用户

- 个人知识工作者：学生、开发者、自媒体作者、研究人员。
- AI 对话重度用户：日常使用 ChatGPT / Claude 进行思考、学习、写作，希望把对话成果沉淀下来的用户。
- 注重数据隐私的个人用户：不希望数据上传到云端，希望数据完全留在本地的用户。

> **产品定位**：纯本地个人工具，不做 C 端用户体系，无需登录注册。启动即用。

### 3.8 AI 智能功能

系统通过统一抽象层接入大语言模型（LLM）和 Embedding 模型，同时支持 **OpenAI 协议** 和 **Anthropic 协议**，便于用户按需切换或扩展。

> **范围边界**：本版本支持 **MCP（Model Context Protocol）** 和 **Skill（技能系统）**，允许 AI 调用外部工具和预定义技能。但不实现自主 Agent 能力（AI 不会主动决策或执行复杂多步骤任务），所有工具调用由用户显式触发或在对话中明确指令。

#### 3.8.3 LLM Provider 抽象设计

```
ILlmProvider (抽象接口)
├── ChatAsync(messages, options) → 流式/非流式响应
├── CompleteAsync(prompt, options) → 文本补全（用于笔记侧 AI 功能）
└── SupportsStreaming → 是否支持 SSE 流式输出

├── OpenAiLlmProvider (OpenAI 协议实现)
│   ├── 兼容 OpenAI API / Azure OpenAI / 第三方兼容服务
│   ├── 使用 chat/completions 端点
│   └── 支持 stream=true SSE 流式
│
└── AnthropicLlmProvider (Anthropic 协议实现)
    ├── 使用 messages 端点
    ├── 支持 Anthropic 原生 stream 事件格式
    └── 仅处理 text 响应，不实现 tool_use / function calling

注意：两种 Provider 的接口均不实现 Agent 能力（tool_use、代码解释器等），
仅返回纯文本补全结果，确保产品边界清晰。
```

#### 3.8.4 Embedding Provider 设计

```
IEmbeddingProvider (抽象接口)
├── EmbedAsync(texts: string[]) → float[][] 向量
└── Dimensions → 向量维度（随模型配置，如 1536 / 1024）

├── OpenAiEmbeddingProvider
│   ├── 使用 OpenAI embeddings 端点
│   └── 支持 text-embedding-3-small / text-embedding-3-large 等
│
└── AnthropicEmbeddingProvider
    └── 使用 Anthropic 兼容的 Embedding 服务
```

#### 3.8.5 向量存储与语义搜索流程

```
笔记保存 → 后台任务 → 调用 Embedding Provider → 生成向量
                                                ↓
                                     存入 note_embeddings (pgvector)
                                                ↓
用户搜索 → 查询关键词同时生成 query embedding → pgvector 相似度检索
                                                ↓
                                   合并全文检索 + 语义检索结果（RRF 融合排序）
```

#### 3.8.6 智能分块（Chunk）算法

为了提升 RAG 检索质量，系统支持多种分块策略，用户可根据需求配置使用哪种策略。所有策略均支持配置使用的 LLM 模型。

**分块策略**

| 策略                               | 优先级 | 描述                                                                                                                            |
| ---------------------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------- |
| **物理分块（Physical Chunking）**  | P0     | 基础保底策略。按固定字符数或 Token 数切分文本，支持配置重叠（overlap）窗口。不依赖 LLM，速度快、成本低                          |
| **智能分块（LLM Chunking）**       | P1     | 使用 LLM 对笔记内容进行分析，按照"总结 + 片段"的结构化方式分块。每个 chunk 包含：段落摘要（便于快速筛选）+ 原文片段（保留细节） |
| **语义分块（Semantic Chunking）**  | P2     | 基于句子/段落的语义相似度自动识别自然边界，在语义转折点切分                                                                     |
| **递归分块（Recursive Chunking）** | P2     | 按标题层级（H1/H2/H3）递归切分，保持文档结构完整性                                                                              |

**物理分块参数**

| 参数          | 说明                                           |
| ------------- | ---------------------------------------------- |
| chunk_size    | 每个分块的最大字符数/Token 数，默认 500        |
| chunk_overlap | 相邻分块的重叠窗口大小，默认 50                |
| separator     | 分隔符优先级列表，如 `\n\n` → `\n` → `.` → ` ` |

**智能分块流程（LLM Chunking）**

```
笔记内容 → 调用 LLM 分析
         ↓
    生成结构化输出：
    {
      "summary": "整篇笔记的核心摘要（200字以内）",
      "chunks": [
        {
          "title": "片段标题",
          "summary": "该片段的要点摘要",
          "content": "原文片段内容",
          "start": 起始位置,
          "end": 结束位置
        }
      ]
    }
         ↓
    每个 chunk 独立生成 Embedding 向量
         ↓
    存入 chunks 表（含 summary_embedding + content_embedding 双向量）
```

**智能分块配置**

| 配置项                | 说明                                                          |
| --------------------- | ------------------------------------------------------------- |
| chunk_llm_model       | 指定用于分块的 LLM 模型（可从已配置的 Provider/Model 中选择） |
| chunk_prompt_template | 自定义分块提示词模板，支持变量替换                            |
| max_chunks_per_note   | 单篇笔记最大分块数，防止过长文档产生过多 chunk                |
| enable_summary        | 是否为每个 chunk 生成独立摘要                                 |

**分块元数据存储**

每个 chunk 记录包含以下信息：
- chunk_id：分块唯一标识
- note_id：所属笔记
- chunk_index：在笔记中的序号
- title：片段标题（智能分块时由 LLM 生成）
- summary：片段摘要（可选）
- content：原文内容
- start_pos / end_pos：在原文中的位置
- summary_embedding：摘要的向量表示（用于粗筛）
- content_embedding：内容的向量表示（用于精排）
- metadata：额外元数据（JSON，如关键词、实体等）

#### 3.8.7 知识图谱

基于笔记内容和分块结果，自动提取实体和关系，构建个人知识图谱，支持可视化浏览和增强检索。

**核心功能**

| 功能           | 优先级 | 描述                                                                |
| -------------- | ------ | ------------------------------------------------------------------- |
| **实体提取**   | P2     | 使用 LLM 从笔记/分块中提取关键实体（人物、概念、技术、组织等）      |
| **关系提取**   | P2     | 识别实体间的关系（属于、相关、依赖、对比等）                        |
| **图谱构建**   | P2     | 将实体和关系存储为图结构，支持增量更新                              |
| **图谱可视化** | P3     | 以交互式节点-边图的形式展示知识网络，支持缩放、拖拽、高亮           |
| **图谱检索**   | P3     | 搜索时结合图谱关系，扩展检索范围（如搜索"React"可关联到"前端框架"） |
| **手动关联**   | P2     | 用户可手动创建/编辑实体和关系，补充自动提取的不足                   |
| **实体消歧**   | P3     | 合并相似实体，解决同名异义问题                                      |

**实体类型**

| 类型                 | 示例                    |
| -------------------- | ----------------------- |
| 概念（Concept）      | React、Transformer、RAG |
| 人物（Person）       | 张三、李四              |
| 组织（Organization） | OpenAI、Microsoft       |
| 技术（Technology）   | PostgreSQL、Docker      |
| 项目（Project）      | Hetu、GPT-4             |
| 自定义（Custom）     | 用户自定义的实体类型    |

**关系类型**

| 关系                  | 示例                          |
| --------------------- | ----------------------------- |
| 属于（belong_to）     | React 属于 前端框架           |
| 相关（related_to）    | Transformer 相关于 注意力机制 |
| 依赖（depends_on）    | Hetu 依赖 PostgreSQL          |
| 包含（contains）      | RAG 包含 检索 + 生成          |
| 对比（compared_with） | SQLite 对比 PostgreSQL        |
| 自定义（custom）      | 用户自定义的关系类型          |

**知识图谱构建流程**

```
笔记保存/更新 → 后台任务
              ↓
         调用 LLM 提取实体和关系
              ↓
    {
      "entities": [
        {"name": "React", "type": "technology", "description": "..."},
        {"name": "前端框架", "type": "concept", "description": "..."}
      ],
      "relations": [
        {"source": "React", "target": "前端框架", "relation": "belong_to"}
      ]
    }
              ↓
         去重 & 实体消歧 → 合并到全局图谱
              ↓
         存储到 graph_entities / graph_relations 表
```

**图谱提取配置**

| 配置项               | 说明                                                 |
| -------------------- | ---------------------------------------------------- |
| extraction_llm_model | 指定用于实体/关系提取的 LLM 模型                     |
| auto_extraction      | 是否在笔记保存时自动触发提取（默认关闭，可手动触发） |
| entity_types         | 需要提取的实体类型列表                               |
| min_confidence       | 实体/关系的最小置信度阈值，低于该值的结果不入库      |

**图谱与 RAG 的结合**

检索时可启用图谱增强：
1. 先通过向量检索找到相关 chunks
2. 从 chunks 中提取实体，在图谱中查找关联实体
3. 将关联实体对应的 chunks 加入候选集
4. 最终排序时给予图谱关联内容一定权重加成

#### 3.8.8 MCP（Model Context Protocol）支持

MCP 是一种让 AI 模型能够访问外部工具和数据源的标准协议。Hetu 支持用户配置 MCP Server，扩展 AI 的能力边界。

**核心功能**

| 功能            | 优先级 | 描述                                                              |
| --------------- | ------ | ----------------------------------------------------------------- |
| MCP Server 管理 | P1     | 用户可添加、编辑、删除 MCP Server 配置                            |
| MCP 协议支持    | P1     | 支持 MCP 协议的 JSON-RPC 通信，包括 tools/list、tools/call 等方法 |
| 工具发现        | P1     | 自动获取 MCP Server 提供的工具列表，展示工具名称、描述、参数定义  |
| 工具调用        | P1     | 在对话中，用户可手动选择工具或让 AI 自动识别并调用合适的工具      |
| 调用结果展示    | P1     | 工具调用的输入参数和返回结果在对话中清晰展示                      |
| 安全沙箱        | P2     | 工具调用在受控环境中执行，防止恶意操作                            |
| 内置 MCP Server | P2     | 预置常用 MCP Server（如文件系统访问、网页抓取等）                 |

**MCP Server 配置**

每个 MCP Server 配置包含：
- 名称：用户自定义标识
- 类型：本地进程（stdio）或远程服务（HTTP/SSE）
- 连接信息：本地命令路径或远程 URL
- 环境变量：传递给 MCP Server 的环境变量（用于存储 API Key 等）
- 启用状态：可随时启用/禁用

**使用场景示例**

- 文件系统访问：AI 可读取/写入本地文件，实现笔记导出、模板填充等
- 数据库查询：AI 可查询外部数据库，获取实时数据
- 网页抓取：AI 可抓取网页内容，用于信息整理
- 第三方服务：通过 MCP 桥接各种 API（如天气、翻译、搜索等）

#### 3.8.7 Skill（技能系统）

Skill 是预定义的可复用能力模块，封装了特定的 AI 工作流。与 MCP 的通用工具调用不同，Skill 更侧重于解决特定场景的问题。

**核心功能**

| 功能         | 优先级 | 描述                                                |
| ------------ | ------ | --------------------------------------------------- |
| Skill 市场   | P2     | 提供官方和社区开发的 Skill，用户可一键安装          |
| Skill 管理   | P1     | 用户可启用/禁用/卸载 Skill                          |
| Skill 配置   | P1     | 部分 Skill 支持自定义参数（如目标语言、输出格式等） |
| Skill 触发   | P1     | 在对话中通过 `/skill-name` 命令或自然语言触发       |
| 自定义 Skill | P3     | 用户可创建自己的 Skill，封装常用的 AI 工作流        |

**内置 Skill 示例**

| Skill 名称 | 功能描述                       |
| ---------- | ------------------------------ |
| 笔记摘要   | 为选中笔记生成结构化摘要       |
| 对话整理   | 将对话内容整理为 Markdown 笔记 |
| 代码解释   | 解释选中的代码片段             |
| 翻译助手   | 翻译文本到指定语言             |
| 语法修正   | 修正文本中的语法错误           |
| 关键词提取 | 从文本中提取关键词             |

**Skill 与 MCP 的区别**

- **Skill**：面向业务场景，封装完整的工作流（如"整理对话为笔记"包含多步骤：提取内容、格式化、保存）
- **MCP**：面向通用工具，提供原子化的能力（如"读取文件"、"执行命令"）
- Skill 内部可以调用 MCP 工具，也可以直接调用 LLM

### 3.9 对话模块（Chat）

对话模块与笔记模块并列，作为产品的一等公民。参考 Cherry Studio 的交互范式，采用「会话组 → 话题 → 消息」三级结构，承载用户与 LLM 的多轮交互。

#### 3.9.1 核心概念

| 概念                             | 说明                                                                                |
| -------------------------------- | ----------------------------------------------------------------------------------- |
| **会话组（ChatGroup）**          | 类比「笔记本」，用于按主题归类对话。例如"论文阅读"、"代码学习"。支持自定义颜色/图标 |
| **话题（Topic）**                | 单次连续的对话会话，隶属于某个会话组。一个话题包含一组有序的消息                    |
| **消息（Message）**              | 单条对话消息，角色为 `user` / `assistant` / `system`                                |
| **上下文窗口（Context Window）** | 每次调用 LLM 时携带的历史消息范围，可设置为"全部"或"最近 N 条"                      |

#### 3.9.2 功能清单

| 功能                        | 优先级 | 描述                                                       |
| --------------------------- | ------ | ---------------------------------------------------------- |
| 会话组 CRUD                 | P0     | 创建、重命名、删除、排序会话组                             |
| 话题 CRUD                   | P0     | 在会话组内创建新话题，重命名、删除、归档                   |
| 发送消息                    | P0     | 用户输入消息，支持换行、粘贴图片（v1.5 支持附件）          |
| 流式输出                    | P0     | 助手消息使用 SSE 流式返回，逐字展示                        |
| 重新生成                    | P0     | 对最近一条助手消息重新生成回答                             |
| 编辑消息                    | P1     | 编辑已发送的 user 消息，重新发起后续对话                   |
| 复制消息                    | P0     | 一键复制消息内容（支持 Markdown 源码 / 渲染结果）          |
| 删除消息                    | P1     | 删除单条消息                                               |
| 切换模型                    | P0     | 每个话题可独立选择使用的 LLM 模型，也可跟随会话组/全局默认 |
| 系统提示词（System Prompt） | P0     | 每个话题可设置系统提示词，预设与自定义均可                 |
| 上下文窗口设置              | P1     | 设置本话题携带的历史消息条数                               |
| 话题分支                    | P2     | 从某条消息分叉出新的对话分支，用于探索不同方向             |
| 话题搜索                    | P1     | 在话题内搜索消息内容                                       |
| 会话组内搜索                | P1     | 在整个会话组内搜索话题和消息                               |
| 多模态消息展示              | P2     | 支持展示模型返回的 Markdown、代码块、LaTeX、Mermaid 图     |

#### 3.9.3 话题列表与交互

- 左侧为会话组列表（类似笔记本树）。
- 中间为当前会话组下的话题列表，可按时间/名称排序。
- 右侧为当前话题的消息区 + 输入区。
- 支持快捷键：`Ctrl+N` 新建话题、`Ctrl+Enter` 发送、`Esc` 停止生成。

### 3.10 对话 → 笔记整理

这是 Hetu 区别于纯对话工具的核心差异化能力：让用户与 AI 的对话成果自动沉淀到笔记知识体系中。

#### 3.10.1 整理方式

| 方式                     | 优先级 | 描述                                                                                                     |
| ------------------------ | ------ | -------------------------------------------------------------------------------------------------------- |
| **一键整理为笔记**       | P0     | 用户点击按钮，调用 LLM 将当前话题的全部消息整理为一篇结构化 Markdown 笔记（含标题、小节、要点、代码块）  |
| **选择片段整理**         | P1     | 用户在话题中勾选若干条消息，仅将这些消息整理为笔记                                                       |
| **自动整理（规则触发）** | P2     | 用户可为会话组设置"自动整理规则"：当话题满足条件（如消息数 ≥ N、持续时长、用户手动标记）时，自动生成笔记 |
| **收藏片段到笔记**       | P1     | 用户在对话中选中一段助手回复，直接追加到指定笔记的末尾（作为引用片段）                                   |

#### 3.10.2 整理流程

```
用户点击"整理为笔记"
   ↓
选择目标笔记本（或新建）
   ↓
选择整理风格：摘要 / 详细 / Q&A / 自定义 Prompt
   ↓
后端组装 Prompt：
   - System: 你是知识整理助手...
   - User: 以下是一段对话内容，请整理为 Markdown 笔记：[消息列表]
   ↓
调用 LLM（chat/completions 或 messages）
   ↓
流式返回 Markdown 草稿
   ↓
用户在编辑器中二次编辑
   ↓
保存为笔记，自动关联标签（可选：由 LLM 推荐）
```

#### 3.10.3 整理选项

- **整理风格**：
  - `摘要式`：提炼要点，精简到 500 字以内
  - `详细式`：保留完整信息，分章节组织
  - `Q&A 式`：将对话改写为问答形式
  - `自定义 Prompt`：用户自己写整理指令
- **目标位置**：指定笔记本 + 可选标签
- **标题生成**：默认由 LLM 根据内容生成，用户可覆盖
- **保留对话原文**：可选将原始对话作为折叠引用附在笔记末尾

### 3.11 预设提示词（Prompt Library）

提供一套可复用的提示词库，既可在对话中使用，也可在笔记 AI 功能中使用。

| 功能         | 优先级 | 描述                                                   |
| ------------ | ------ | ------------------------------------------------------ |
| 系统预设     | P0     | 内置常用提示词：翻译、润色、摘要、代码解释、头脑风暴等 |
| 用户自定义   | P1     | 用户创建、编辑、删除自己的提示词                       |
| 分类管理     | P1     | 按分类（工作、学习、写作、编程...）组织提示词          |
| 变量插槽     | P2     | 提示词中支持 `{{变量}}` 占位符，使用时填写             |
| 在对话中使用 | P0     | 新建话题时可选择预设作为 System Prompt                 |
| 在笔记中使用 | P1     | 选中笔记文本后，调用预设提示词进行转换                 |
| 导入/导出    | P2     | 支持 JSON 格式导入导出提示词库                         |

## 4. 非功能需求

### 4.1 性能

- 首屏加载时间 ≤ 2 秒（在 4G 网络下）。
- 笔记列表滚动流畅，支持虚拟滚动（笔记数 > 500）。
- 搜索响应时间 ≤ 500ms（数据量 < 10 万条）。
- 自动保存不阻塞用户输入。

### 4.2 可用性

- 全年可用性目标 ≥ 99.5%。
- 关键操作具备离线草稿能力（后续扩展 PWA）。
- 界面支持简体中文，后续可扩展国际化。

### 4.3 安全

- 密码使用 PBKDF2/Argon2 哈希存储。
- 所有 API 通过 JWT 鉴权，Token 有效期 24 小时，支持 Refresh Token。
- 输入内容做 XSS 过滤，Markdown 渲染使用可信库并消毒。
- 数据库连接字符串等敏感配置不提交到代码仓库。
- HTTPS 强制，HSTS 启用。

### 4.4 可扩展性

- 后端采用分层架构，便于后续接入缓存（Redis）、消息队列、对象存储。
- 前端组件化设计，便于新增编辑器模式与插件。

### 4.5 兼容性

- 支持 Chrome、Edge、Firefox、Safari 最新两个主版本。
- 后端部署于 Linux/Windows 容器。

## 5. 技术架构

### 5.1 总体架构

系统采用前后端分离架构，分为四层：

| 层级           | 技术栈                  | 说明                                            |
| -------------- | ----------------------- | ----------------------------------------------- |
| **前端展示层** | React + TypeScript      | 前端 SPA，Vite 构建，React Query 管理服务端状态 |

| **AI 服务层**  | 自研统一抽象层          | 支持 OpenAI 协议 / Anthropic 协议，可配置切换   |
| **数据存储层** | PostgreSQL 15+          | 主数据存储 + pgvector 向量扩展                  |
| **外部服务**   | OpenAI / Anthropic API  | 外部 LLM 和 Embedding 服务                      |

调用链路：用户 → 前端 SPA → ASP.NET Core API → AI 服务层 → 外部 LLM/Embedding API；同时 API → PostgreSQL（主存储 + 向量存储）。

| AI - LLM       | 自研统一抽象层              | 支持 OpenAI 协议 / Anthropic 协议，可配置切换 |

## 6. 数据库设计

### 6.1 实体关系

**笔记模块**
- 笔记本包含多个子笔记本（自引用 1:N）
- 笔记本包含多篇笔记（1:N）
- 笔记可关联多个标签（N:M）
- 笔记可有多个历史版本（1:N）
- 笔记对应一条 Embedding 向量记录（1:1，可选）

**对话模块**
- 会话组包含多个话题（1:N）
- 话题包含多条消息（1:N）
- 话题关联一个 LLM 模型（N:1，可空，空则跟随默认）
- 话题可关联一条系统提示词预设（N:1，可空）

**AI 配置**
- 可配置多个 AI Provider（1:N）
- 一个 Provider 下可配置多个 Model（1:N）
- 可配置多个 MCP Server（1:N）
- 可安装多个 Skill（1:N）

### 6.2 核心数据实体

#### 6.2.1 笔记模块

| 实体                           | 关键字段                                                                                         | 说明                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------------ | ---------------------------------------------------- |
| **笔记本（Notebooks）**        | Id, ParentId, Name, SortOrder, CreatedAt, UpdatedAt                                              | 支持无限层级嵌套                                     |
| **笔记（Notes）**              | Id, NotebookId, Title, Content, IsDeleted, IsFavorite, IsPinned, DeletedAt, CreatedAt, UpdatedAt | 支持软删除、收藏、置顶                               |
| **标签（Tags）**               | Id, Name, Color, CreatedAt                                                                       | 全局唯一                                             |
| **笔记标签关联（NoteTags）**   | NoteId, TagId                                                                                    | 多对多关联                                           |
| **历史版本（NoteVersions）**   | Id, NoteId, Title, Content, CreatedAt                                                            | 保留历史快照                                         |
| **笔记向量（NoteEmbeddings）** | NoteId, Embedding, Model, CreatedAt, UpdatedAt                                                   | 依赖 pgvector 扩展（PG模式）或简化存储（SQLite模式） |

#### 6.2.2 AI 配置

| 实体                         | 关键字段                                                                | 说明                               |
| ---------------------------- | ----------------------------------------------------------------------- | ---------------------------------- |
| **AI 供应商（AiProviders）** | Id, ProviderType, Name, ApiKey, BaseUrl, IsEnabled                      | ProviderType: openai/anthropic     |
| **AI 模型（AiModels）**      | Id, ProviderId, ModelId, DisplayName, Purpose, IsDefault, ContextWindow | Purpose: chat/embedding/completion |

#### 6.2.3 对话模块

| 实体                                      | 关键字段                                                                                                                                            | 说明                        |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------- |
| **会话组（ChatGroups）**                  | Id, Name, Description, Color, Icon, SortOrder, CreatedAt, UpdatedAt                                                                                 | 类比笔记本，按主题归类对话  |
| **话题（ChatTopics）**                    | Id, GroupId, Title, ModelId, SystemPromptPresetId, CustomSystemPrompt, ContextWindowSize, IsArchived, IsAutoOrganizeEnabled, AutoOrganizeNotebookId | 单次连续对话会话            |
| **消息（ChatMessages）**                  | Id, TopicId, Role, Content, ParentId, ModelId, TokensUsed, LatencyMs, CreatedAt                                                                     | Role: user/assistant/system |
| **预设提示词（PromptPresets）**           | Id, Category, Name, Content, Variables, IsBuiltIn, SortOrder, CreatedAt, UpdatedAt                                                                  | 支持变量插槽                |
| **对话整理到笔记关联（ChatToNoteLinks）** | Id, TopicId, NoteId, OrganizeStyle, MessageIds, CreatedAt                                                                                           | 溯源对话→笔记的整理记录     |

#### 6.2.4 扩展模块

| 实体                         | 关键字段                                                                                           | 说明                             |
| ---------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------- |
| **MCP Server（McpServers）** | Id, Name, Type, ConnectionConfig, EnvironmentVariables, IsEnabled, SortOrder, CreatedAt, UpdatedAt | Type: stdio/http-sse             |
| **Skill（Skills）**          | Id, Name, Description, Category, IsBuiltIn, IsEnabled, Config, SortOrder, CreatedAt, UpdatedAt     | 预定义的可复用 AI 能力模块       |
| **应用设置（AppSettings）**  | Key, Value, UpdatedAt                                                                              | 存储显示名称、主题、数据库配置等 |

### 6.3 索引策略

| 索引类型           | 说明                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| **全文搜索索引**   | SQLite 使用 FTS5 虚拟表；PostgreSQL 使用 tsvector GIN 索引                       |
| **向量相似度索引** | SQLite 使用简化存储（后续可接入向量扩展）；PostgreSQL 使用 pgvector HNSW/IVFFlat |
| **常用查询索引**   | 笔记更新时间、笔记本删除状态、话题创建时间等高频查询场景                         |

## 8. 安全设计

- **输入校验**：后端 FluentValidation + 前端 Zod，防止非法输入。
- **XSS 防护**：Markdown 渲染使用 DOMPurify 消毒；禁止用户直接插入 `<script>`。
- **API Key 加密**：AI Provider 的 API Key 使用 DPAPI / 应用密钥环加密存储，不以明文落库。
- **MCP 安全沙箱**：MCP 工具调用在受控环境中执行，防止恶意操作。
- **数据本地化**：默认使用 SQLite 本地存储，数据完全保留在用户设备上。
- **审计日志**：记录关键操作（删除、导出、AI 调用）到日志系统。

**系统质量**
- [ ] 数据默认存储在本地 SQLite，数据文件可备份和迁移。
- [ ] 前端在 Chrome/Edge/Firefox/Safari 最新两个版本正常可用。
